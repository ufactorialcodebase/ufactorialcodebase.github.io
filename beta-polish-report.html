Backend blocker (predates this commit)

RLS regression on conversation_sessions. Every POST /api/chat/greeting returns 500 with new row violates row-level security policy for table "conversation_sessions" (Postgres 42501). The local test backend is on the ISS-231 fix branch (7c3e241), which is code-identical to main for the artifact path — so this is unrelated to ISS-231. Same failure observed for other test users (e2e_mc04a_*) in the same log window, so it's not pratikcpednekar_test2-specific.

What this blocks

  • Every playwright test that seeds via chat: Items 1, 4, 5-Topics (beforeAll seed), 6, 7 (beforeAll seed).
  • People sort test — the shared beforeAll in 05-sort-controls.spec.js seeds entities via chat; when seed times out, the People test's data is empty and the SortToggle doesn't mount.
  • Item 8 — depends on entities/topics existing from prior seeds; no new data means the world graph is empty.

What this does NOT block

  • The 2 shipped commits are code-review-verifiable — small, focused, forward-compatible changes.
  • Batch-1 items already merged; no live regression on prod (Railway is on main).
  • Vitest suite (format-utils boundary tests etc.) — unaffected by backend state.

Suggested fix path

  • Trace where conversation_sessions gets its INSERT — likely a Session adapter using a Supabase client that lost its SET LOCAL app.current_user_id or was migrated to a non-service-role client.
  • Migration 011_enable_rls.sql defines conversation_sessions_user_isolation policy on USING (user_id = current_setting('app.current_user_id', true)) — the setting has to be set on the connection before every session insert or the RLS check fails.
  • Verify the failure reproduces on prod / Railway — if prod is fine, the regression is local-env or SUPABASE_ENV=test-only. If prod is also failing, this is a live outage.
